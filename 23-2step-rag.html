2-Step RAG — Better Retrieval with Query Rewriting & Reranking

⏱ ~20 min💻 Runnable code🔍 Precision retrieval

What you'll learn: Why naive vector search often retrieves the wrong chunks, and three techniques that fix it: query rewriting (before retrieval), hybrid search (semantic + keyword), and cross-encoder reranking (after retrieval).

🔎
Analogy: the librarian's two jobs

A good librarian doesn't just search by your exact words — they first clarify what you actually need ("are you looking for the history or the science?"), then they search, and then they rank the results by relevance to your actual goal, not just keyword frequency. 2-step RAG does the same: rewrite the query to what the retriever should find, then rerank the results by true relevance before passing them to the model.

The Problem with Naive RAG

❌ Naive RAG fails when…

The user's question is ambiguous ("how do I do it?"), uses conversational language that doesn't match document terminology, asks a multi-part question, or asks a follow-up that depends on prior conversation context. The embedding of "how do I do it?" matches nothing useful.

✅ 2-Step RAG solves this by…

Using an LLM to rewrite the user's question into a precise search query before retrieval, then reranking the retrieved chunks by actual relevance before passing them to the final generation step. Two small LLM calls dramatically improve the quality of what the model sees.

Technique 1: Query Rewriting

pythonquery_rewriting.py
from langchain.chat_models import init_chat_model
from langchain_core.messages import HumanMessage, SystemMessage
from pydantic import BaseModel

llm = init_chat_model("openai:gpt-4o-mini")

# ── Simple query rewriting ─────────────────────────────────────────────────────
class RewrittenQuery(BaseModel):
    standalone_query: str
    search_queries: list[str]  # Multiple search angles

rewriter = llm.with_structured_output(RewrittenQuery)

def rewrite_query(user_question: str, chat_history: list = None) -> RewrittenQuery:
    """Convert a conversational question into precise search queries."""
    history_text = ""
    if chat_history:
        history_text = "\n".join([
            f"User: {m['user']}\nAssistant: {m['assistant']}"
            for m in chat_history[-3:]  # last 3 turns
        ])

    result = rewriter.invoke([
        SystemMessage(f"""Given a conversation and a follow-up question, rewrite the question into:
1. A standalone question that captures full intent (even if it's a follow-up)
2. 2-3 search query variations to maximize retrieval coverage

Conversation history:
{history_text or "(none)"}"""),
        HumanMessage(user_question)
    ])
    return result

# Example
history = [
    {"user": "Tell me about our sick leave policy", "assistant": "Employees get 10 sick days per year..."}
]
rewritten = rewrite_query("Can I carry those over?", history)
print(rewritten.standalone_query)
# → "Can unused sick leave days be carried over to the next year?"
print(rewritten.search_queries)
# → ["sick leave carryover policy", "unused PTO rollover", "sick days carry forward"]

# Use all search queries and merge results
def multi_query_retrieve(rewritten: RewrittenQuery, retriever, top_k=6):
    """Run multiple query variations and deduplicate results."""
    seen_ids = set()
    all_docs = []

    # Include the standalone query + all search variations
    queries = [rewritten.standalone_query] + rewritten.search_queries

    for query in queries:
        docs = retriever.invoke(query)
        for doc in docs:
            doc_id = hash(doc.page_content[:100])
            if doc_id not in seen_ids:
                seen_ids.add(doc_id)
                all_docs.append(doc)

    return all_docs[:top_k]

dense_retriever = vectorstore.as_retriever(search_kwargs={"k": 5})

bm25_retriever = BM25Retriever.from_documents(chunks, k=5)

)

# Use it exactly like a regular retriever
docs = ensemble_retriever.invoke("GDPR Article 17 right to erasure policy")
# Dense finds conceptually similar docs about data deletion
# BM25 finds docs that literally mention "Article 17" or "right to erasure"

Technique 3: Cross-Encoder Reranking

pythonreranking.py
from langchain.retrievers.document_compressors import CrossEncoderReranker
from langchain.retrievers import ContextualCompressionRetriever
from langchain_community.cross_encoders import HuggingFaceCrossEncoder

# ── Why rerank? ────────────────────────────────────────────────────────────────
# Vector search ranks by embedding similarity — fast but approximate.
# A cross-encoder reads the FULL query + document pair and scores relevance
# directly. Much more accurate, but slower. Run it on the top-20 candidates
# to select the best 4 to pass to the model.

# ── Using a HuggingFace cross-encoder (free, runs locally) ────────────────────
cross_encoder = HuggingFaceCrossEncoder(
    model_name="cross-encoder/ms-marco-MiniLM-L-6-v2"
)

reranker = CrossEncoderReranker(model=cross_encoder, top_n=4)

# Wrap the base retriever with the reranker
# Flow: base_retriever returns 20 docs → reranker scores all 20 → top 4 returned
compression_retriever = ContextualCompressionRetriever(
    base_compressor=reranker,
    base_retriever=ensemble_retriever.as_retriever(search_kwargs={"k": 20}),
)

docs = compression_retriever.invoke("what is the employee vacation policy?")
# Returns only the 4 most relevant chunks, scored by actual query-document relevance

# ── Cohere Reranker (API-based, higher quality) ────────────────────────────────
from langchain_cohere import CohereRerank

cohere_reranker = CohereRerank(
    model="rerank-english-v3.0",
    top_n=4,
)

cohere_retriever = ContextualCompressionRetriever(
    base_compressor=cohere_reranker,
    base_retriever=ensemble_retriever,
)

The Full 2-Step Pipeline

pythontwo_step_pipeline.py
from langchain.chat_models import init_chat_model
from langchain_core.messages import SystemMessage, HumanMessage

final_llm = init_chat_model("openai:gpt-4o-mini")

def two_step_rag(question: str, chat_history: list = None) -> str:
    """
    Full 2-step RAG pipeline:
    1. Rewrite query → multi-query retrieve → rerank
    2. Generate answer from reranked context
    """
    # ── Step 1a: Rewrite ──────────────────────────────────────────────────────
    rewritten = rewrite_query(question, chat_history)
    print(f"Rewritten query: {rewritten.standalone_query}")

    # ── Step 1b: Multi-query retrieve with hybrid search ──────────────────────
    candidate_docs = multi_query_retrieve(rewritten, ensemble_retriever, top_k=20)
    print(f"Retrieved {len(candidate_docs)} candidates")

    # ── Step 1c: Rerank to top 4 ──────────────────────────────────────────────
    # Cross-encoder scores each (query, document) pair independently
    ranked = cross_encoder.score([
        (rewritten.standalone_query, doc.page_content)
        for doc in candidate_docs
    ])
    top_docs = [doc for _, doc in sorted(zip(ranked, candidate_docs), reverse=True)[:4]]
    print(f"Reranked to top {len(top_docs)} docs")

    # ── Step 2: Generate ──────────────────────────────────────────────────────
    context = "\n\n---\n\n".join([
        f"[Source: {doc.metadata.get('source', 'unknown')}]\n{doc.page_content}"
        for doc in top_docs
    ])

    response = final_llm.invoke([
        SystemMessage(f"""Answer the question using ONLY the context below.
Cite sources with [Source: filename] notation.
If the context doesn't contain the answer, say so clearly.

Context:
{context}"""),
        HumanMessage(rewritten.standalone_query),
    ])

    return response.content

# Use it as an agent tool
@tool
def search_knowledge_base(question: str, runtime: ToolRuntime) -> str:
    """Search company knowledge base with intelligent query rewriting and reranking.

    Args:
        question: The question to answer from the knowledge base.
    """
    return two_step_rag(question)
⚠️
Gotcha: reranking adds latency and cost

Running a cross-encoder on 20 documents adds ~200–500ms of latency. For user-facing applications, cache reranked results for common queries, or use the Cohere API which is already optimized. If latency is critical and you can't cache, skip the cross-encoder and use hybrid search alone — it gets you 80% of the benefit. Only add reranking when retrieval quality is genuinely a problem.

📋 IT Support Knowledge Bot

An IT helpdesk bot suffers from poor retrieval — when users ask "my laptop screen went dark", it retrieves docs about screen brightness settings instead of the power management troubleshooting guide. With query rewriting, the ambiguous query becomes "laptop display turns off power management troubleshooting". With hybrid search, it catches both semantic matches and exact ticket references like "JIRA-2847". With reranking, the actual troubleshooting steps move to the top. First-contact resolution rate improves from 42% to 71%.

✅
Key Takeaways

Three techniques stack on basic RAG: query rewriting (LLM rewrites vague questions into precise search terms before retrieval), hybrid search (combine dense semantic + sparse BM25 keyword search via EnsembleRetriever), and cross-encoder reranking (score query-document pairs more accurately after retrieval via ContextualCompressionRetriever). Use all three together for the best quality. Add incrementally starting with query rewriting, which gives the biggest bang for the buck.
