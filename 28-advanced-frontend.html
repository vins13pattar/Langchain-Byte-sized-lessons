Advanced Frontend — Optimistic UI, Persistence & Error Recovery

⏱ ~20 min💻 Runnable code⚡ Production UX

What you'll learn: How to build production-quality agent UIs: optimistic updates so the UI feels instant, conversation history persistence across page reloads, error recovery with retry logic, and loading state management for long agent runs.

📮
Analogy: postal tracking vs. guaranteed delivery

A naïve mail system only tells you when the package arrives or fails. A good postal service shows "Accepted", "In transit", "Out for delivery" — you see progress at every step and know what to do if it fails. Advanced frontend patterns do this for agent runs: show intermediate states, handle failures gracefully with retry, and restore state after browser refresh.

Optimistic Updates

typescripthooks/useOptimisticChat.ts
// Show the user's message immediately, before the server confirms it
// This makes the UI feel instant even with slow network or backend startup latency

import { useOptimistic } from 'react'  // React 19+ / Next.js 14+

export function useChatWithOptimistic() {
  const [confirmedMessages, setConfirmedMessages] = useState([])

  // useOptimistic: adds a temporary version before the real update arrives
  const [optimisticMessages, addOptimistic] = useOptimistic(
    confirmedMessages,
    (state, newMessage: Message) => [...state, newMessage]
  )

  const send = async (text: string) => {
    const tempId = `optimistic-${Date.now()}`

    // 1. Show user message immediately (optimistic)
    addOptimistic({
      id: tempId,
      role: 'user',
      content: text,
      status: 'pending',
    })

    // 2. Add placeholder assistant message
    addOptimistic({
      id: `${tempId}-response`,
      role: 'assistant',
      content: '',
      status: 'streaming',
    })

    // 3. Actually send and update state when confirmed
    try {
      await streamFromAgent(text, (token) => {
        // Update assistant content as tokens arrive
        setConfirmedMessages(prev =>
          prev.map(m => m.id === `${tempId}-response`
            ? { ...m, content: m.content + token, status: 'streaming' }
            : m
          )
        )
      })

      // Mark as confirmed
      setConfirmedMessages(prev =>
        prev.map(m => m.id === `${tempId}-response`
          ? { ...m, status: 'done' }
          : m
        )
      )
    } catch (err) {
      // Show error state on failure
      setConfirmedMessages(prev =>
        prev.map(m => m.id === `${tempId}-response`
          ? { ...m, content: 'Something went wrong. Please try again.', status: 'error' }
          : m
        )
      )
    }
  }

  return { messages: optimisticMessages, send }
}

Persistence Across Page Reloads

typescripthooks/usePersistedChat.ts
// Two-layer persistence:
// 1. sessionStorage — fast, clears when tab closes (good for active session)
// 2. Backend API — durable, loads from checkpointer history

export function usePersistedChat(threadId: string) {
  const [messages, setMessages] = useState(() => {
    // On mount: try to restore from sessionStorage first (instant)
    const cached = sessionStorage.getItem(`chat-${threadId}`)
    if (cached) return JSON.parse(cached)
    return []
  })

  const [isLoadingHistory, setIsLoadingHistory] = useState(messages.length === 0)

  // Load full history from backend on first mount (if nothing cached)
  useEffect(() => {
    if (messages.length > 0) {
      setIsLoadingHistory(false)
      return
    }

    fetch(`/api/sessions/${threadId}/history`)
      .then(r => r.json())
      .then(({ messages: history }) => {
        setMessages(history)
        sessionStorage.setItem(`chat-${threadId}`, JSON.stringify(history))
        setIsLoadingHistory(false)
      })
      .catch(() => setIsLoadingHistory(false))
  }, [threadId])

  // Persist to sessionStorage on every update
  const updateMessages = useCallback((updater: (prev: Message[]) => Message[]) => {
    setMessages(prev => {
      const next = updater(prev)
      sessionStorage.setItem(`chat-${threadId}`, JSON.stringify(next))
      return next
    })
  }, [threadId])

  return { messages, isLoadingHistory, updateMessages }
}

Error Recovery with Retry

typescriptlib/retryFetch.ts
// Exponential backoff retry for transient failures (503, timeout, etc.)
async function retryFetch(
  url: string,
  options: RequestInit,
  maxRetries = 3,
  onRetry?: (attempt: number, delay: number) => void
): Promise {
  let lastError: Error | null = null

  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      const res = await fetch(url, options)

      // Don't retry 4xx errors (client errors — retrying won't help)
      if (res.status >= 400 && res.status < 500) return res

      // Retry on 5xx (server errors) or network failures
      if (res.ok) return res

      throw new Error(`Server error: ${res.status}`)
    } catch (err) {
      lastError = err as Error

      if (attempt < maxRetries) {
        const delay = Math.min(1000 * 2 ** attempt + Math.random() * 500, 10_000)
        onRetry?.(attempt + 1, delay)
        await new Promise(r => setTimeout(r, delay))
      }
    }
  }

  throw lastError
}

// In the UI:
// const [retryInfo, setRetryInfo] = useState<{ attempt: number; delay: number } | null>(null)
//   — shows "Retrying in 2s (attempt 2/3)"

// Reconnecting SSE: browser EventSource auto-reconnects,
// but when using fetch-based SSE, handle reconnection manually:
async function streamWithReconnect(url: string, options: RequestInit, onChunk: (data: string) => void) {
  let retries = 0
  const maxRetries = 3

  while (retries <= maxRetries) {
    try {
      const res = await fetch(url, options)
      const reader = res.body!.getReader()
      const decoder = new TextDecoder()

      while (true) {
        const { done, value } = await reader.read()
        if (done) return
        onChunk(decoder.decode(value))
      }
    } catch {
      retries++
      if (retries > maxRetries) throw new Error('Connection failed after retries')
      await new Promise(r => setTimeout(r, 1500 * retries))
    }
  }
}

⚠️
Gotcha: SSE connections drop after 30s on many platforms

Vercel serverless functions, Cloudflare Workers, and some load balancers time out long-lived connections at 30 seconds. For agents that take longer, either use a background job + polling pattern (client polls /api/status/{job_id} every 2s), or deploy the streaming backend on a platform without function timeouts (Railway, Render, Fly.io). Next.js edge functions have no timeout limit but require the edge runtime.

⚖️ Contract Analysis Agent

Analyzing a 50-page contract takes 2–3 minutes. The UI shows a live progress timeline: "Uploading document → Parsing PDF → Extracting clauses → Checking against playbook → Identifying risks → Generating summary". Each step is driven by a runtime.stream_writer.emit() call from the agent (custom events via astream_events). The user sees exactly what's happening, never wonders if the app froze, and gets an estimated completion time. If the connection drops, the retry logic reconnects and the progress UI picks up from the last event.

✅
Key Takeaways

Use useOptimistic (React 19+) to show user messages instantly before server confirmation. Persist chat to sessionStorage for fast restores, and fall back to the backend history API on cold load. Implement exponential backoff retry for transient network errors. For long-running agents (>10s), show a live step timeline driven by custom streaming events. Watch out for SSE timeout limits on serverless platforms — prefer non-timeout runtimes for heavy agent workloads.
