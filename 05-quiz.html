Module 05 Quiz

Tools — Creating

🧠 Multiple Choice (5 questions · pass = 4/5)

Q1. What is the simplest way to create a LangChain tool?

@tool is the decorator API — annotate any Python function and LangChain automatically builds the tool schema from the signature and docstring.

Q2. What does the docstring on a @tool function become?

The docstring IS the tool description the LLM sees. A good docstring directly determines whether the agent calls the tool at the right time.

Q3. How do you create an async tool?

Simply define the function with async def under @tool. LangChain automatically detects async functions and calls them with await.

Q4. What happens if a tool raises an exception during execution?

By default LangChain catches tool exceptions and returns the error message as a string result. The agent sees the error and can decide how to handle it.

Q5. What must tools return for the LLM to process the result?

Tool outputs must be strings because they become ToolMessage content. Return JSON-serialized strings for structured data.

🎤 Interview Questions
What makes a good tool description? How does a bad description hurt agent performance?▼
A good description is precise about WHEN to use the tool, WHAT inputs it expects, and WHAT it returns. Example: 'Search the company knowledge base for product documentation. Input: a natural language question. Returns: relevant text excerpts.' A bad description ('Search stuff') causes the agent to call the wrong tool, miss the right tool, or provide incorrect arguments — directly harming answer quality.
How would you give a tool access to the current user's identity without passing it through the LLM?▼
Use InjectedToolArg to mark the user_id parameter as runtime-injected (not filled by the LLM), then pass it via config['configurable'] using ToolRuntime. The LLM never sees this parameter in the schema — it's invisible to the model but accessible to the tool function at runtime.
Describe the difference between synchronous and asynchronous tools and when to use each.▼
Sync tools block the event loop during execution — fine for fast operations. Async tools (async def) are non-blocking — essential for I/O-bound operations like database queries, HTTP calls, or file reads. In production, always make tools that call external services async. Sync tools in an async agent run in a thread pool automatically, but async is preferred for performance.
