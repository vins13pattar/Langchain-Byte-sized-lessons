Module 32 Quiz

Deploy & Observe

Q Multiple Choice (5 questions, pass = 4/5)

Q1. What uvicorn flag enables multi-process deployment for handling concurrent requests?

--workers 4 starts 4 separate uvicorn processes. Each handles requests independently. Use CPU_COUNT or CPU_COUNT*2 workers. Each worker loads the agent separately.

Q2. What does --timeout-keep-alive 120 do?

--timeout-keep-alive 120 keeps idle SSE connections open for 120s. Default is 5s - too short for streaming agents where pauses between tokens may exceed 5s.

Q3. What Python library provides structured JSON logging?

structlog outputs structured JSON logs: {timestamp, level, event, user_id, latency_ms, ...}. Machine-readable, queryable in CloudWatch/Datadog, far more useful than plain text logs.

Q4. Which Prometheus metric should you track to detect infinite agent loops?

llm_tokens_total tracks cumulative token consumption. An agent stuck in a loop burns tokens continuously - a sudden spike in tokens_per_request signals a looping agent.

Q5. Which platform has SSE limitations due to serverless function timeouts?

Vercel serverless functions time out after 30s. Long agent responses are cut off. Use Vercel's streaming edge functions, or deploy to Railway/Render/Fly.io/Cloud Run for unlimited SSE duration.

A Interview Questions
Walk through the Docker and uvicorn configuration for a production agent deployment.▼
Dockerfile: FROM python:3.12-slim. COPY requirements.txt . RUN pip install -r requirements.txt --no-cache-dir. COPY . . CMD ['uvicorn', 'main:app', '--host', '0.0.0.0', '--port', '8000', '--workers', '4', '--timeout-keep-alive', '120']. docker-compose.yml: app service with .env_file, postgres service for checkpointer, health check on /health. Production: run behind nginx or a load balancer. Set workers = 2*CPU_count. Use --log-level warning to reduce log noise. Add --access-log for request audit trail.
What Prometheus metrics would you implement and what alerts would you set up?▼
Metrics: agent_requests_total (counter, labels: status, model), agent_latency_seconds (histogram, p50/p95/p99), llm_tokens_total (counter, labels: model, type), tool_calls_total (counter, labels: tool_name, status), guardrail_triggers_total (counter, labels: type). Alerts: p99 latency > 30s, error rate > 5%, tokens/request > 50k (looping agent), tool error rate > 10%, guardrail trigger spike (potential attack). Dashboard: request rate, latency distribution, token burn rate, top errors.
How do you implement zero-downtime deployments for a stateful agent service?▼
Challenge: agents use checkpointers (database) and may have active SSE streams. Strategy: (1) Use a persistent checkpointer (Postgres) - all state is in DB, not process memory. (2) Rolling deploy: bring up new containers before killing old ones (Kubernetes rolling update, Railway/Fly.io rolling). (3) Drain: on SIGTERM, stop accepting new requests but finish active SSE streams (uvicorn handles this). (4) Health checks: new container must pass /health before traffic routes to it. (5) Database migrations: run before deploy, backward-compatible. (6) Feature flags: roll out new agent behavior gradually.
