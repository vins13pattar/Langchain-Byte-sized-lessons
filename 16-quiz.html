Module 16 Quiz

Context Engineering

🧠 Multiple Choice (5 questions · pass = 4/5)

Q1. What is the 'lost in the middle' problem?

Research shows LLMs recall information from the beginning and end of context better than the middle. Long RAG contexts with relevant info buried in the middle perform poorly.

Q2. What is few-shot prompting?

Few-shot prompting includes examples that show the model exactly what format and reasoning you expect — dramatically improves consistency on structured tasks.

Q3. What is the primary purpose of RAG (Retrieval-Augmented Generation)?

RAG retrieves relevant documents from a knowledge base and injects them into the prompt — giving the LLM access to current or proprietary knowledge without retraining.

Q4. Where should the most important context be placed in a long prompt?

Due to the lost-in-the-middle effect, critical instructions and most relevant context should appear at the start or end of the prompt, not buried in the middle.

Q5. What is 'context compression' used for?

Context compression (e.g., LLMChainExtractor) summarizes or filters retrieved documents to keep only the most relevant sentences before injecting them — reducing token usage and improving signal-to-noise.

🎤 Interview Questions
Compare RAG vs. fine-tuning for injecting domain knowledge. When would you choose each?▼
RAG: Injects knowledge at inference time from a retrieval step. Best for: frequently updated knowledge, large knowledge bases, explainable citations, private data. No retraining needed. Fine-tuning: Bakes knowledge into model weights. Best for: stable knowledge, style/format changes, when retrieval latency is unacceptable. Choose RAG first — it's faster to build, easier to update, and provides citations. Fine-tune only when RAG quality is insufficient or latency requirements can't be met.
How does few-shot prompting improve agent tool call accuracy?▼
Without examples, the LLM must infer the expected format from the schema alone. With 2-3 examples of (user query → correct tool call + args), the model learns: which tool to use for which phrasing, how to format edge-case arguments, and when NOT to use a tool. For tools with subtle distinctions (e.g., 'search_products' vs 'search_inventory'), few-shot examples are often the difference between 70% and 95% correct tool selection.
What strategies reduce the lost-in-the-middle effect in RAG pipelines?▼
(1) Limit retrieved chunks — top-3 instead of top-10. (2) Rerank retrieved chunks by relevance and place the most relevant first AND last (sandwich pattern). (3) Context compression — extract only the relevant sentences from each chunk. (4) Map-reduce: summarize each chunk independently, then summarize the summaries — no long middle context. (5) Use a model with strong long-context performance (Claude, Gemini 1.5) for long-context tasks.
