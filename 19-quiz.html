Module 19 Quiz

Human-in-the-Loop

🧠 Multiple Choice (5 questions · pass = 4/5)

Q1. How do you pause an agent mid-execution for human review?

In v1, HumanInTheLoopMiddleware pauses matching tool calls by calling LangGraph's interrupt() primitive. State is saved to the checkpointer and execution halts until you resume with a decision.

Q2. How do you resume a paused agent with a human decision?

Resume by invoking with a Command(resume=...) carrying the decision list, the same thread_id, and version="v2". Each decision is approve, reject, edit, or respond.

Q3. What checkpointer is required for HITL patterns?

HITL requires state to persist while waiting for human input - which could be minutes or hours. InMemorySaver is lost on restart.

Q4. For custom approval logic, where do you call interrupt()?

Call interrupt() inside wrap_tool_call before invoking handler(request) — the tool hasn't run yet. interrupt() returns the human's decision when you resume with Command(resume=...).

Q5. How do you change a tool's arguments before it runs?

The edit decision supplies corrected tool name/args; the modified tool then executes. (For deeper graph edits, agent.update_state() is the lower-level fallback.)

🎤 Interview Questions
Walk through the full HITL flow for an agent that needs approval before sending an email.▼
(1) Agent (with HumanInTheLoopMiddleware(interrupt_on={"send_email": True})) proposes the send_email call. (2) Middleware calls interrupt(); invoke with version="v2" returns a GraphOutput whose .interrupts hold the pending action. (3) State saved to AsyncPostgresSaver. (4) Backend stores (thread_id, action) in an approval queue. (5) Email to approver with Approve/Reject links. (6) Approve → agent.ainvoke(Command(resume={"decisions": [{"type": "approve"}]}), config, version="v2"). (7) Tool executes and sends. Reject → resume with {"type": "reject", "message": ...}; the agent responds without sending.
How would you implement a HITL timeout - auto-reject if no human responds within 24 hours?▼
Store the interrupt timestamp alongside thread_id. A scheduled job queries pending approvals older than 24h. For each: agent.ainvoke(Command(resume={"decisions": [{"type": "reject", "message": "Auto-rejected: approval timeout"}]}), config, version="v2"). The agent handles the rejection gracefully. Alert the original requester that their action was auto-rejected with an explanation.
What are the UX implications of HITL and how do you handle them in production?▼
UX challenge: user sends a request then waits indefinitely. Solutions: (1) Immediate acknowledgment: 'Your request requires approval - you will be notified.' (2) Status polling endpoint: GET /api/request/{id}/status returns 'pending'/'approved'/'rejected'. (3) Push notification on approval. (4) Show pending approvals in UI. (5) Allow user to cancel while pending. (6) Clear SLAs: 'Approvals typically happen within 1 business hour.'
