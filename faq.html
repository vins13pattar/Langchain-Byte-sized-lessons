⚙️ Setup & Installation
What exactly do I need to install?

Two things: the core package and at least one provider package for the model you want.

pip install -U langchain langchain-openai python-dotenv

langchain gives you create_agent, tools, and middleware. langchain-openai connects to OpenAI. Prefer Claude? pip install langchain-anthropic. Gemini? langchain-google-genai. See Module 02.

I get ModuleNotFoundError: No module named 'langchain_openai'

The core langchain package doesn't bundle providers. Install the provider package separately: pip install langchain-openai (note: the import name uses an underscore, langchain_openai, but the pip name uses a hyphen, langchain-openai).

Also confirm you installed into the same environment your script runs in — activate your venv first: source .venv/bin/activate.

How do I set my API key safely?

Never hardcode it. Use an environment variable or a .env file:

OPENAI_API_KEY=sk-proj-...
# optional but recommended
LANGSMITH_TRACING=true
LANGSMITH_API_KEY=lsv2_pt_...

Then load it at the top of your script with from dotenv import load_dotenv; load_dotenv(). LangChain reads the provider key automatically from the environment.

Which Python version do I need?

Python 3.10+ is recommended for LangChain v1 (3.9 may work for parts but newer typing features are used widely). Always work inside a virtual environment so dependencies stay isolated.

📦 Imports & the v0 → v1 confusion
Where does create_agent come from? Old tutorials use different imports

In LangChain v1 the canonical import is:

from langchain.agents import create_agent
from langchain.chat_models import init_chat_model

Older (v0) tutorials use from langgraph.prebuilt import create_react_agent — that's the predecessor. create_agent replaced create_react_agent. If a tutorial imports create_react_agent, it's pre-v1.

langchain vs langchain_core vs langchain.agents — which do I use?

Rule of thumb for v1:

  • langchain.agents → create_agent, AgentState
  • langchain.chat_models → init_chat_model
  • langchain.tools → tool, ToolRuntime
  • langchain.agents.middleware → all middleware classes and decorators (before_model, wrap_tool_call, …)
  • langchain.agents.structured_output → ToolStrategy, ProviderStrategy
  • langgraph.types → Command, interrupt

langchain_core is the low-level foundation. langchain.messages and langchain_core.messages both work for HumanMessage/AIMessage etc.; the docs prefer langchain.messages.

Where do middleware decorators import from?

Everything middleware-related — classes and decorators — lives in one place in v1:

from langchain.agents.middleware import (
    AgentMiddleware, before_model, after_model,
    wrap_model_call, wrap_tool_call, dynamic_prompt,
    SummarizationMiddleware, HumanInTheLoopMiddleware,
)

Older paths like langchain.middleware or ...middleware.decorators are not the v1 path. See Module 13.

I followed an old tutorial and nothing works — what changed in v1?

Common v0 → v1 shifts: create_react_agent → create_agent; chains (LLMChain, RetrievalQA) are replaced by composing agents + tools + middleware; HITL moved to HumanInTheLoopMiddleware + the interrupt() primitive (no more raising NodeInterrupt). When in doubt, check the changelog / migration guide and prefer code that matches the current quickstart.

🧠 Models
What is a "model string" like openai:gpt-4o-mini?

It's a "provider:model-name" identifier that init_chat_model (and create_agent) understands. The provider prefix tells LangChain which integration to use. Examples: openai:gpt-4o-mini, anthropic:claude-sonnet-4-6, google_genai:gemini-2.5-flash-lite.

Gotcha: dropping the provider prefix (just "gpt-4o") can fail — keep the prefix. See Module 03.

init_chat_model() vs ChatOpenAI() — which should I use?

init_chat_model("openai:gpt-4o-mini") is the portable, provider-agnostic way — swap providers by changing one string. ChatOpenAI(model="gpt-4o-mini") is the direct class, useful when you need OpenAI-specific options. For most agents, prefer init_chat_model.

I get an AuthenticationError / 401

The provider key is missing or wrong. Check: (1) the env var name matches the provider (OPENAI_API_KEY, ANTHROPIC_API_KEY, …), (2) load_dotenv() runs before you create the model, (3) the key has no trailing spaces/quotes, (4) your account has billing/credits enabled.

Do I have to use OpenAI?

No. LangChain's value is a standard interface across providers. Install the matching package and change the model string — Anthropic, Google, AWS Bedrock, Azure, Ollama (local), and many more are supported. Your agent code stays the same.

What does temperature do?

It controls randomness. temperature=0 makes output (near-)deterministic — best for tool-using agents and structured output where you want consistency. Higher values (0.7–1.0) add creativity, useful for brainstorming or varied prose.

🔧 Tools
Why isn't my tool being called?

The model decides whether to call a tool based on its name, docstring, and type hints. The usual culprits:

  • Missing type hints — they generate the schema; without them the model can't call the tool correctly.
  • Weak docstring — the docstring is the description the model reads. Be specific: say when to use it.
  • Vague name — use clear snake_case verbs like search_database.
from langchain.tools import tool

@tool
def get_weather(city: str) -> str:
    """Get the current weather for a city. Use for any weather question."""
    return f"Sunny in {city}"

See Module 05.

How does a tool read conversation state or user info?

Add a runtime: ToolRuntime parameter — it's injected automatically and hidden from the model. Through it you get runtime.state (short-term memory), runtime.context (immutable per-invocation data like a user ID), and runtime.store (long-term memory).

from langchain.tools import tool, ToolRuntime

@tool
def whoami(runtime: ToolRuntime) -> str:
    """Return the current user id."""
    return runtime.context.user_id

Gotcha: config and runtime are reserved argument names. See Module 06.

🤖 Agents & the loop
What is the "agent loop" / ReAct loop?

The cycle the agent runs: reason (model decides what to do) → act (call a tool) → observe (tool result is added to messages) → reason again. It repeats until the model answers with no further tool calls. See the interactive diagram in Module 02 and depth in Module 07.

create_agent vs LangGraph vs Deep Agents — when to use which?
  • Deep Agents — batteries-included (planning, virtual filesystem, subagents). Start here for maximum capability with minimal setup.
  • LangChain create_agent — a minimal, highly customizable harness. Use when you want fine-grained control over model, tools, and middleware. This course teaches this.
  • LangGraph — the low-level orchestration framework underneath, for mixing deterministic and agentic steps. Reach for it when a plain loop isn't enough.
How do I read the agent's answer?

The final reply is the last message:

result = agent.invoke({"messages": [{"role": "user", "content": "Hi"}]})
print(result["messages"][-1].content)

The v1 docs also show .content_blocks, which returns content as a structured list (text, tool calls, reasoning, images) — useful for multimodal/reasoning models. For plain text, .content is fine.

My agent loops forever / burns tokens — how do I cap it?

Set a recursion_limit so a misbehaving agent can't loop indefinitely, and add a model-call-limit or token-budget middleware as a cost safety net. Set the limit based on your most complex legitimate task, not higher. See Module 14.

💾 Memory & state
Why does my agent forget everything between turns?

By default each invoke is stateless. To remember, give the agent a checkpointer and pass a thread_id:

from langgraph.checkpoint.memory import InMemorySaver
agent = create_agent(model, tools, checkpointer=InMemorySaver())

cfg = {"configurable": {"thread_id": "user-42"}}
agent.invoke({"messages": [...]}, config=cfg)  # same thread_id remembers

See Module 09.

What is a thread_id?

It's the conversation key. All turns sharing a thread_id share message history via the checkpointer. Use one per user/conversation. It's also how a paused (human-in-the-loop) run is found and resumed later.

InMemorySaver vs a real database?

InMemorySaver is perfect for prototyping but loses everything on restart. For production (and any human-in-the-loop flow where a human may take minutes/hours), use a persistent checkpointer like AsyncSqliteSaver (dev) or AsyncPostgresSaver (prod). See Module 12.

⚡ Streaming & async
invoke vs ainvoke vs stream — what's the difference?
  • invoke — run once, get the full result (blocking).
  • ainvoke — same, but async (use inside async apps like FastAPI).
  • stream / astream — yield partial output as it's produced (tokens, state updates) for a live UI.

See Module 10 and Module 11.

What are stream modes (values / updates / messages)?

values = full state after each step; updates = just what changed; messages = token-by-token LLM output (for typing-effect UIs). You can combine them, e.g. stream_mode=["updates", "messages"]. See Module 10.

📚 RAG
When do I need RAG instead of just a tool?

Use RAG when answers must be grounded in your documents/knowledge that the model wasn't trained on. You embed and index your content, then retrieve the most relevant chunks at query time and feed them to the model. If the data is reachable by a simple API call, a plain tool may be enough. See Module 22.

What are embeddings and a vector store, in one line each?

Embeddings turn text into numeric vectors capturing meaning. A vector store indexes those vectors so you can find the chunks most similar to a question. The retriever is the query interface over the store. See Module 22.

🔍 Debugging & production
How do I see what my agent is actually doing?

Enable LangSmith tracing — set LANGSMITH_TRACING=true and LANGSMITH_API_KEY in your environment. No code changes needed; every run is logged with the full message chain, tool calls, arguments, latency, and tokens. Free tier at smith.langchain.com. See Module 29.

For quick local inspection, print the whole history: for m in result["messages"]: print(type(m).__name__, m.content).

How do I keep costs under control?

Use a cheaper model for simple steps (and a stronger one only when needed via dynamic model selection middleware), cap iterations with recursion_limit, trim/summarize long histories (SummarizationMiddleware), enable prompt caching for large static prompts, and watch token usage in LangSmith. See Module 16 and Module 32.

🗺️ Learning path
I'm brand new — where do I start and in what order?

Go in order: 01 (What is LangChain) → 02 (build a working agent) → the Core Components (03–08) → then memory, middleware, and the advanced parts. The course deliberately teaches components before customization and working code before theory. Do the quiz after each module to lock it in.

Do I need to know LangGraph first?

No. create_agent is built on LangGraph but you don't need LangGraph knowledge to start. Pick it up later (it's referenced in Modules 07, 19, 20) when you need custom orchestration beyond the standard agent loop.
