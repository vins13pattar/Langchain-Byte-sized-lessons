Tools — Advanced Patterns

⏱ ~22 min💻 Runnable code🔧 ToolRuntime diagram

What you'll learn: How to inject runtime context into tools (state, user context, store, streaming), how tools can update agent state using Command returns, how to use ToolNode for custom workflows, and advanced error handling patterns.

The Problem: Tools Need Context

In Module 05, every tool was a pure function — input in, result out. But production tools often need more: they need to know which user is running the agent, access shared state, write streaming progress updates, or even change the agent's conversation history.

ToolRuntime is LangChain's solution — a dependency injection system that passes context into tools at call time, without the model needing to supply it as arguments.

💡
The Hotel Concierge Analogy

Imagine a hotel concierge (the tool) who helps guests (the model). The guest asks "book me a restaurant for tonight." The concierge doesn't need the guest to tell them the hotel's Michelin Guide subscription, the loyalty program database, or the manager's emergency contact — they have access to all of that through the hotel's systems (ToolRuntime). The guest just asks; the context is injected.

ToolRuntime: Injecting Context

Declare a runtime parameter (reserved name) in your tool to receive runtime context:

pythontool_runtime.py
from langchain.tools import tool
from langchain.tools import ToolRuntime  # the injected context object

@tool
def get_my_profile(runtime: ToolRuntime) -> dict:
    """Get the current user's profile information.

    Returns the authenticated user's name, email, and role.
    No arguments needed — uses the current session context.
    """
    # runtime.context contains whatever you passed in as context_schema
    user_id = runtime.context.get("user_id")
    email = runtime.context.get("email")

    # Look up in database (example)
    return {
        "user_id": user_id,
        "email": email,
        "role": "admin",  # fetched from DB in real code
    }

# The agent is created with a context_schema that defines what context means
from langchain.agents import create_agent
from langchain.chat_models import init_chat_model

agent = create_agent(
    model=init_chat_model("openai:gpt-4o"),
    tools=[get_my_profile],
    context_schema={"user_id": str, "email": str},
)

# At invocation time, you supply the context — the model never sees it
result = agent.invoke(
    {"messages": [{"role": "user", "content": "What's my email?"}]},
    context={"user_id": "usr_123", "email": "alice@example.com"},
)
print(result["messages"][-1].content)
# → Your email is alice@example.com.

All ToolRuntime Fields — Click to Explore

runtime.context
dict — from context_schema
User-supplied invocation context. Good for: user identity, tenant ID, locale, permissions.
@tool
def get_user_data(query: str, runtime: ToolRuntime) -> dict:
    """Fetch user-specific data."""
    tenant = runtime.context["tenant_id"]
    user = runtime.context["user_id"]
    return db.query(f"SELECT * FROM {tenant}.users WHERE id=?", [user])
runtime.state
AgentState — current conversation state
Read the agent's current state (messages, custom fields). Read-only view — use Command to update.
@tool
def check_history(runtime: ToolRuntime) -> str:
    """Check if this topic was discussed before."""
    messages = runtime.state["messages"]
    # Count how many turns we've had
    human_msgs = [m for m in messages if m.type == "human"]
    return f"This is turn {len(human_msgs)} of the conversation."
runtime.store
BaseStore — long-term memory
Read and write to the agent's persistent store. Shared across sessions. Great for user preferences, learned facts.
@tool
async def remember_preference(
    key: str, value: str, runtime: ToolRuntime
) -> str:
    """Save a user preference for future sessions."""
    user_id = runtime.context["user_id"]
    namespace = ("preferences", user_id)
    await runtime.store.aput(namespace, key, {"value": value})
    return f"Remembered: {key} = {value}"
runtime.stream_writer
Callable — emit custom stream events
Push progress updates to the frontend while the tool is still running. Perfect for long operations.
@tool
async def analyze_large_file(
    path: str, runtime: ToolRuntime
) -> str:
    """Analyze a large file and stream progress."""
    writer = runtime.stream_writer
    lines = open(path).readlines()
    for i, line in enumerate(lines):
        if i % 1000 == 0:
            writer({"progress": i/len(lines), "status": f"Analyzed {i} lines"})
        # ... process line ...
    return f"Analysis complete: {len(lines)} lines processed."

The Command Return Type

By default, a tool returns a value and the agent continues. But sometimes a tool needs to update the agent's state — add messages, change custom fields, or even route the agent to a different node. That's what Command is for:

pythoncommand_return.py
from langchain.tools import tool
from langgraph.types import Command
from langchain_core.messages import ToolMessage

@tool
def save_and_respond(
    important_info: str,
    runtime: ToolRuntime
) -> Command:
    """Save important info and add a system note to the conversation.

    Args:
        important_info: The information to save and note.
    """
    # Return a Command that BOTH provides a tool result AND updates state
    return Command(
        # This is what gets added to the conversation as the tool result
        update={
            "messages": [
                ToolMessage(
                    content=f"Saved: {important_info}",
                    tool_call_id=runtime.tool_call_id,
                )
            ],
            # Also update a custom state field
            "saved_items": runtime.state.get("saved_items", []) + [important_info],
        }
    )

# With Command, tools can:
# 1. Add arbitrary messages (not just the tool result)
# 2. Update custom AgentState fields
# 3. Route to specific graph nodes (goto parameter)
# 4. Combine all of the above

# Routing example — send agent to a specific next step
@tool
def escalate_to_human(reason: str, runtime: ToolRuntime) -> Command:
    """Escalate the conversation to a human agent."""
    return Command(
        update={
            "messages": [ToolMessage(
                content=f"Escalating: {reason}",
                tool_call_id=runtime.tool_call_id,
            )],
            "needs_human": True,
            "escalation_reason": reason,
        },
        goto="human_handoff_node",  # route to a specific graph node
    )

ToolNode: Tools in Custom Workflows

When building custom LangGraph workflows (beyond create_agent()), you use ToolNode to handle tool execution:

pythontool_node.py
from langgraph.prebuilt import ToolNode, tools_condition
from langgraph.graph import StateGraph, START, END
from langchain_core.messages import HumanMessage
from langchain.chat_models import init_chat_model
from langchain.tools import tool
from typing import TypedDict, Annotated
from langgraph.graph.message import add_messages

@tool
def search_web(query: str) -> str:
    """Search the web."""
    return f"Results for: {query}"

class AgentState(TypedDict):
    messages: Annotated[list, add_messages]

model = init_chat_model("openai:gpt-4o").bind_tools([search_web])
tool_node = ToolNode([search_web])  # handles tool execution

def call_model(state: AgentState):
    return {"messages": [model.invoke(state["messages"])]}

# Build the graph
graph = StateGraph(AgentState)
graph.add_node("agent", call_model)
graph.add_node("tools", tool_node)
graph.add_edge(START, "agent")
# tools_condition: go to "tools" if model called a tool, else END
graph.add_conditional_edges("agent", tools_condition)
graph.add_edge("tools", "agent")  # loop back after tool execution

app = graph.compile()

result = app.invoke({"messages": [HumanMessage("Search for LangChain news")]})
💡
ToolNode vs create_agent()

create_agent() uses ToolNode internally. You only need to use ToolNode directly when building custom LangGraph workflows that need non-standard routing logic — like looping back to different nodes depending on which tool was called, or running tools in parallel.

Async Tools

For I/O-bound operations (API calls, database queries), async tools avoid blocking:

pythonasync_tool.py
import asyncio
import httpx
from langchain.tools import tool

# Async tool — define with async def
@tool
async def fetch_stock_price(ticker: str) -> dict:
    """Get the current stock price for a ticker symbol.

    Args:
        ticker: The stock ticker (e.g., AAPL, MSFT, GOOGL).
    """
    async with httpx.AsyncClient() as client:
        response = await client.get(
            f"https://api.example.com/stock/{ticker}",
            timeout=10.0,
        )
        data = response.json()
        return {"ticker": ticker, "price": data["price"], "change": data["change"]}

# Must use async invoke with async tools
from langchain.agents import create_agent
from langchain.chat_models import init_chat_model

agent = create_agent(
    model=init_chat_model("openai:gpt-4o"),
    tools=[fetch_stock_price],
)

# In an async context:
async def main():
    result = await agent.ainvoke({
        "messages": [{"role": "user", "content": "What's Apple's stock price?"}]
    })
    print(result["messages"][-1].content)

asyncio.run(main())
⚠️
Gotcha: runtime is injected, not schema'd

The runtime: ToolRuntime parameter is completely invisible to the model — it won't appear in the tool schema, and the model won't try to fill it. It's purely a server-side injection. If you accidentally name another parameter runtime, LangChain will try to inject ToolRuntime into it and your tool will break in confusing ways.

⚠️
Gotcha: Command requires tool_call_id

When returning a Command that includes a ToolMessage, you MUST include the tool_call_id that links back to the AIMessage that invoked the tool. Get it from runtime.tool_call_id. Missing this will cause a validation error from most providers.

Real-World Use Case

📊 Analytics Agent with Live Progress

A data analysis agent runs long-running SQL queries. Each tool uses runtime.context to get the user's database credentials and tenant ID (never exposed to the model), runtime.stream_writer to push "Query running: 45% complete" updates to the frontend in real time, and Command returns to store intermediate results in a custom AgentState field. The model sees only the final formatted summary — the messy progress is hidden in the stream.

✅
Key Takeaways

Use runtime: ToolRuntime to inject context (user identity, state, store, streaming) without the model needing to know. Use Command returns when a tool needs to update agent state or route to a different node. Use ToolNode directly only in custom LangGraph workflows. Async tools require async invoke throughout.
