Module 18 Quiz

Guardrails

🧠 Multiple Choice (5 questions · pass = 4/5)

Q1. Where is the best place to implement input guardrails?

@before_model fires before each LLM call - ideal for input guardrails. You can inspect, modify, or reject the input before it reaches the LLM.

Q2. What is 'LLM-as-judge' for guardrails?

LLM-as-judge calls a fast, cheap LLM (e.g., haiku) to classify inputs/outputs. It catches nuanced policy violations that regex cannot.

Q3. What is the main risk of overly aggressive guardrails?

False positives block legitimate users. A guardrail that rejects 5% of valid queries is a significant UX problem. Balance security with precision.

Q4. How do you scrub PII from tool outputs before they reach the LLM?

In wrap_tool_call: output = yield; return scrub_pii(output). The tool runs (yield), you receive its output, scrub it, and return the clean version to the LLM.

Q5. What is the recommended approach for handling guardrail violations?

Tell the user what happened. Vague or silent rejections frustrate users and provide no path to resolution.

🎤 Interview Questions
Design a multi-layer guardrail system for an enterprise agent.▼
Layer 1 (@before_model): regex-based fast block for obvious violations (SSNs, credit cards). ~0ms latency. Layer 2 (@before_model): LLM-as-judge call (haiku) for nuanced policy violations. ~100ms. Layer 3 (@after_model): output validation - check response doesn't contain confidential data. Layer 4 (@wrap_tool_call after yield): PII scrub on tool outputs. Log all violations with user_id and triggering layer - never log raw PII.
How would you measure and improve guardrail precision and recall?▼
Precision = true blocks / all blocks (low = too many false positives). Recall = true blocks / all violations (low = too many missed violations). Build a labeled dataset of (input, should_block) pairs from real traffic and adversarial test cases. Measure current guardrail metrics. To improve: tune thresholds, add examples to the LLM-as-judge prompt, or use a fine-tuned classifier. Re-evaluate monthly.
What is the difference between synchronous and asynchronous guardrails?▼
Synchronous: guardrail blocks until check completes. Required for blocking guardrails where you must know the verdict before proceeding. Asynchronous: guardrail runs in parallel - user gets response immediately; violation is logged post-hoc. Use for audit logging, shadow-mode testing new guardrails. Most input guardrails must be synchronous. Output monitoring can often be async.
