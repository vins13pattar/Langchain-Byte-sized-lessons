Module 22 Quiz

RAG Fundamentals

🧠 Multiple Choice (5 questions · pass = 4/5)

Q1. What does RAG stand for?

Retrieval-Augmented Generation: retrieve relevant documents from a knowledge base, then augment the LLM's generation with that retrieved context.

Q2. What is text splitting used for in RAG?

LLMs have context limits. Documents must be split into chunks (~500-1000 tokens) before embedding. Chunk size affects retrieval quality.

Q3. What do embeddings represent?

An embedding is a vector that encodes semantic meaning of text. Similar meanings produce similar vectors - enabling semantic similarity search.

Q4. What method converts a vector store into a retriever?

vectorstore.as_retriever(search_kwargs={'k': 4}) returns a Retriever. Call retriever.invoke('query') to get the top-k most similar documents.

Q5. What does chunk_overlap do in text splitting?

chunk_overlap (e.g., 200 tokens) repeats the end of one chunk at the start of the next. Without overlap, sentences spanning boundaries would be split - losing context.

🎤 Interview Questions
Walk through building a basic RAG pipeline from scratch.▼
(1) Load documents: DirectoryLoader or WebBaseLoader. (2) Split: RecursiveCharacterTextSplitter(chunk_size=1000, chunk_overlap=200). (3) Embed: OpenAIEmbeddings(). (4) Store: Chroma.from_documents(chunks, embeddings). (5) Retrieve: retriever = vectorstore.as_retriever(search_kwargs={'k': 4}). (6) Tool: @tool def search_kb(query): return retriever.invoke(query). (7) Agent: create_agent(model, tools=[search_kb]).
What is the difference between semantic search and keyword search?▼
Keyword search (BM25, TF-IDF): finds documents containing exact query words. Fast, interpretable, good for precise matches ('AWS S3 bucket policy'). Semantic search: finds documents with similar meaning even without keyword overlap ('cloud storage permissions'). Better for natural language queries and synonyms. Best practice: hybrid search combining BM25 + vector - outperforms either alone.
How does chunk size affect RAG retrieval quality?▼
Too small (100 tokens): chunks lose surrounding context - retrieved chunks may be incomplete. Too large (2000 tokens): chunks cover multiple topics - less precise, one chunk matches many queries. Sweet spot: 500-1000 tokens for text, 100-200 overlap. Code: smaller chunks (100-300) since code is denser. Tune empirically using retrieval precision/recall metrics on your specific corpus.
