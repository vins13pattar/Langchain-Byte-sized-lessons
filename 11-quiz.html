Module 11 Quiz

Event Streaming

🧠 Multiple Choice (5 questions · pass = 4/5)

Q1. What version of astream_events should you use?

Always use astream_events(input, version='v2'). v1 is deprecated — v2 has better performance and a cleaner event schema.

Q2. Which event type fires for each token chunk during LLM streaming?

on_chat_model_stream fires for each token chunk. Access the text via event['data']['chunk'].content.

Q3. Which event fires when a tool starts execution?

on_tool_start fires when a tool begins executing. It includes the tool name and input — useful for showing 'Searching...' UI indicators.

Q4. How do you dispatch a custom event from inside a tool?

await adispatch_custom_event('name', data) fires a custom event that appears in the astream_events stream with kind='on_custom_event'.

Q5. How do you filter astream_events to only LLM token events?

Iterate the async generator and check event['event'] == 'on_chat_model_stream' — there's no built-in filter param, you filter manually in the loop.

🎤 Interview Questions
Walk through how you'd build a real-time agent progress UI using astream_events.▼
async for event in agent.astream_events(input, version='v2'): if event['event'] == 'on_chat_model_stream': yield token to frontend (streaming text). elif event['event'] == 'on_tool_start': send {type: 'tool_start', name: event['name']} to show 'Using [tool]...' UI. elif event['event'] == 'on_tool_end': send {type: 'tool_end', output: event['data']['output']} to update tool result. elif event['event'] == 'on_custom_event': handle your custom progress events. Use SSE to send each event to the frontend.
How would you use custom events to report progress from a long-running tool?▼
Inside the tool: await adispatch_custom_event('progress', {'step': 'Searching database', 'pct': 30}). The frontend listens for on_custom_event with kind='progress' and updates a progress bar. This is superior to just logging — it makes internal tool progress observable without coupling the tool to any UI code. Works with any async tool.
What are the performance trade-offs of streaming vs. waiting for a full response?▼
Streaming: Better perceived performance (lower TTFT), user can stop reading early, supports abort. Costs: more complex infrastructure (SSE/WebSocket), buffering complexity, harder to retry failed streams, can't cache streaming responses. Full response: Simpler, cacheable, retryable, lower infrastructure overhead. Rule of thumb: always stream for interactive chat; use full responses for batch processing, webhooks, and cached responses.
